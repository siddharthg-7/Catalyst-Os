1. Introduction

1.1 Purpose

Catalyst OS is an AI command-center for early-stage founders. A founder states a goal in plain language; a CEO Orchestrator breaks it into sub-tasks, routes them to specialist agents (Finance, Hiring, Growth, Legal, Investment/Operations), which visibly collaborate, produce recommendations, and surface high-stakes actions for founder approval before anything executes.

1.2 Scope for this hackathon

A working, deployed slice proving goal → orchestration → specialist agent → approval → dashboard update, end to end, with 1–2 agents fully real and the rest visibly stubbed but well-dressed. Judging rewards a focused working system over a broad broken one.

1.3 Definitions

  • Orchestrator / CEO agent — router interpreting founder intent and delegating to specialists.
  • Specialist agent — a domain-scoped AI worker (Finance, Hiring, Growth, Legal, Investment).
  • Approval Gate — queue where high-stakes outputs wait for founder confirmation.
  • Health Score — composite number summarizing startup status across domains.
  • Memory context — recent activity-log entries injected into an agent's prompt so responses reference prior decisions.

2. Overall Description

2.1 Product perspective

A single desktop-first web app: command box (text, optional voice), dashboard of executive cards, approval queue, and a "boardroom" visualization layer that sits between command input and dashboard update.

2.2 User classes

Founder — single role, single demo account. No multi-tenant auth needed.

2.3 Operating environment

  • Frontend: React/Next.js on Vercel.
  • Backend: FastAPI (or Node/Express) on Railway/Render.
  • Database: managed Postgres (Supabase/Neon).
  • LLM: one hosted API (Claude or Gemini), one key. No local/fine-tuned models.
  • Voice (optional): browser-native Web Speech API only. No external voice vendor.

2.4 Design constraints

36 hours, 5 people, no budget for paid third-party APIs (Plaid, Stripe, Deel, DocuSign, ElevenLabs, Deepgram, Crunchbase, Apollo — all ⚪). No Celery/Redis/LangGraph/pgvector — single backend process is sufficient at demo scale.

3. Functional Requirements

4.1 Landing & Onboarding 🟢 MVP-36H

  • FR-1.1: Landing page, product one-liner, "Get Started" / "Demo Startup" CTAs.
  • FR-1.2: Onboarding form: startup name, industry, stage, team size, monthly budget, one-line idea.
  • FR-1.3: On submit, generates a CompanyContext record in Postgres, redirects to Dashboard.
  • FR-1.4 🟡: File upload accepted for "financial statement"; seeded data used regardless of contents.

4.2 CEO Orchestrator 🟢 MVP-36H

  • FR-2.1: Accepts free-text founder goal via command box.
  • FR-2.2: LLM call classifies intent into finance, hiring, growth (stub), legal (stub), investment (stub).
  • FR-2.3: Routes to the corresponding agent handler.
  • FR-2.4: Triggers the Executive Collaboration Engine animation (§4.7) instead of a generic spinner.

4.3 Finance Agent 🟢 MVP-36H

  • FR-3.1: Compute burn rate and runway deterministically from seeded transaction data.
  • FR-3.2: Pass computed numbers to the LLM for a founder-facing narrative summary.
  • FR-3.3: Expose a callable check_budget(amount) function usable by other agents.
  • FR-3.4: Return output in structured schema.

4.4 Hiring Agent 🟢 MVP-36H

  • FR-4.1: Calls Finance's check_budget() before responding to a hiring request.
  • FR-4.2: Generates a job description via LLM.
  • FR-4.3: Ranks a fixed set of 5–10 seeded mock candidates via a simple keyword/skill-overlap score.
  • FR-4.4: Drafts one outreach email per top candidate (held as a draft, never sent).

4.5 Approval Center 🟢 MVP-36H

  • FR-5.1: Every "executing" action lands in a pending queue instead of auto-executing.
  • FR-5.2: Founder can Approve / Reject / Edit.
  • FR-5.3: Approving updates Health Score and Activity Timeline, marks item EXECUTED.

4.6 Explainable AI 🟢 MVP-36H

  • FR-6.1: Every agent recommendation returns structured JSON: recommendation, reasons: [string], confidence: float, source: string.
  • FR-6.2: Dashboard renders reasons as an expandable "Why?" section.

4.7 Executive Collaboration Engine 🟢 MVP-36H

Purpose: Replace the generic loading spinner with a visualization of which departments are "active" while a request processes — this is the single highest demo-impact, lowest-cost item in this revision.
  • FR-7.1: On command submission, UI shows a boardroom-style layout.
  • FR-7.2: Departments relevant to the classified intent illuminate in sequence with contextual status text.
  • FR-7.3: For the two real agents (Finance, Hiring), status text reflects actual pipeline stage.
  • FR-7.4: Sequence resolves to "Execution Plan Ready" before the dashboard updates.

4.8 Startup Memory Engine (Scoped) 🟢 MVP-36H

  • FR-8.1: Every founder command and agent output is written to activity_log.
  • FR-8.2: When an agent runs, the last 5–10 relevant activity_log rows are included in prompt context.
  • FR-8.3: Agent references this history in its narrative.

4.9 Voice Interface (Browser-Native, Optional) 🟡 SIMULATED

  • FR-9.1: Founder may submit commands via text (primary) or voice (optional).
  • FR-9.2: Uses the browser's built-in SpeechRecognition API.
  • FR-9.4: Optional spoken response via SpeechSynthesis API.

4.10 Executive Inbox / Daily Brief 🟢 MVP-36H

  • FR-10.1: On load, render summary block: Health Score delta, pending approvals, recent actions.
  • FR-10.2: Content generated from existing queries.

4.11 Executive Meeting Summary 🟢 MVP-36H

  • FR-11.1: Aggregate structured outputs from all agents touched by the current request into a single summary view.

6. System Architecture (Hackathon-Realistic)

Founder (Text, optional Voice via Web Speech API) │ ▼ CEO Orchestrator (LLM intent classify) │ ▼ Executive Collaboration Engine (visual layer) │ ┌────────────────┼────────────────┬─────────┬───────────┐ ▼ ▼ ▼ ▼ ▼ Finance Agent Hiring Agent Growth (stub) Legal(stub) Investment(stub) │ │ └── check_budget() real call ──┘ │ ▼ Structured JSON output (per agent) │ ▼ Executive Meeting Summary (aggregation) │ ▼ Approval Queue (Postgres) │ Founder approves │ ▼ Dashboard: Health Score + Timeline + Executive Inbox update │ ▼ activity_log row written → feeds Memory context (§4.8) on next relevant agent call

Stack: Next.js + Tailwind frontend; FastAPI backend; Postgres (Supabase/Neon); one LLM provider with structured JSON output mode (temperature 0 for Finance, 0.7 for Growth/Hiring drafting); browser-native Web Speech API for optional voice.

8. Demo Script (target ~3 minutes)

  1. Founder opens dashboard — Executive Inbox shows "Good morning" brief with Health Score and pending items.
  2. Founder types (or speaks): "We need to hire a backend engineer."
  3. Executive Collaboration Engine animates — Hiring and Finance nodes illuminate, Hiring visibly consults Finance via check_budget().
  4. Executive Meeting Summary appears: one line each from Finance and Hiring, CEO recommendation.
  5. Founder expands "Why?" on the Hiring recommendation — sees reasons + confidence.
  6. Item lands in Approval Center. Founder clicks Approve.
  7. Dashboard updates: Health Score changes, Activity Timeline logs it, Executive Inbox reflects the new state.
  8. (If stable) Founder asks a direct Finance question — real computed burn/runway narrative, referencing memory context ("since your last hiring decision...").
  9. Close on the Production Roadmap slide: full 5-pillar vision, real integrations, voice vendor, semantic memory — framed as next steps.

9. Explicitly Out of Scope for the Hackathon Build

ElevenLabs/Deepgram voice vendors · LangGraph · Celery/Redis · pgvector/full RAG · fine-tuned local LLM · Plaid/Stripe/GitHub/Deel/DocuSign/Slack/Google Workspace integrations · multi-tenant auth · desktop push notifications · Auditor Agent self-correction loop · real email sending · semantic/long-term memory beyond current-session activity log.

These aren't rejected as bad ideas — they're the real production roadmap. They're excluded here because building any one of them well would consume the entire 36 hours on its own, at the cost of the working demo.
